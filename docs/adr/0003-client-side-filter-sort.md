# 0003. クライアント側フィルタ / ソート

- ステータス:採用(中規模まで)
- 決定日:2026-06-01

## 文脈

クライアント / 求人 / 応募 / 面接 / お知らせ などの一覧表示で、ユーザは検索 / ステータス絞り込み / ソートをしたい。
2 つのパターンがある:

1. **サーバー側**:URL パラメータ + 都度クエリ。状態を共有しやすい、巨大データに強い。
2. **クライアント側**:全件取得 → JS でフィルタ / ソート。レスポンス即座、UX 軽い。

Maira の現在の規模感(1 エージェント企業あたり数十〜数百件のクライアント / 求人):サーバー往復は過剰。

## 決定

中規模(〜数百件)まではクライアント側でフィルタ / ソートする。

- 純関数 `applyXxxFilterSort` を `lib/{module}/filter-sort.ts` に集約
- 入力配列を破壊しない(slice → sort)
- 全フィルタ軸を 1 つの options オブジェクトで受ける
- テスト容易(副作用ゼロ)

代表例:

- [lib/clients/filter-sort.ts](../../lib/clients/filter-sort.ts)
- [lib/jobs/filter-sort.ts](../../lib/jobs/filter-sort.ts)
- [lib/applications/filter-sort.ts](../../lib/applications/filter-sort.ts)

UI 側は `useMemo` で結果配列を計算、`useState` で各軸を保持。

## 結果

得たもの:

- 検索 / ソート時のレスポンスがほぼ瞬時(ネットワーク往復なし)
- 純関数なのでテストが書きやすい
- フィルタ軸の追加が UI 側だけで完結

諦めたもの / 課題:

- 千件超になるとブラウザの初期描画が遅くなる
- 全件取得分の通信量が増える
- 暗号化フィールド(`encrypted_*`)で検索したいケースに対応できない

## 移行計画

「全件 + JS フィルタ」が不適切になる兆候:

- `getClientRecordsTotalCount(orgId)` で 1000 件超を検出
- ダッシュボードに「サーバページネーション切替を検討」バナー(実装済)

そのとき切り替える先:

- `listClientRecordsPaginated(orgId, { limit, offset })`(基盤実装済)
- URL パラメータ ベースでフィルタ条件を保持
- 必要なら全文検索インデックス(pg_trgm 等)

## 代替案

1. **サーバ側ページネーション + フィルタ from day 1**:却下。MVP 規模では過剰。実装コストとレスポンス遅延が見合わない。
2. **ハイブリッド**(クライアント数で動的切替):採用候補。閾値超えで自動切替。実装計画には入れたが未着手。
