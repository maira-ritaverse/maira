# 0001. 暗号化戦略

- ステータス:採用
- 決定日:2026-05-20

## 文脈

ユーザの個人情報(履歴書・職務経歴書・対応履歴・推薦コメント・面談所感等)を Supabase に保存する。
平文で保存すると次のリスクがある:

- DB ダンプ流出時に直接読まれる
- Supabase 側の万一の不正アクセスで全件露呈
- 運営者(エージェント企業)が他社の機密を見られる事故

求職者個人にも「自分のデータは運営者からも見えない」という安心感を提供したい。

## 決定

サーバーサイド AES-256-GCM 暗号化を採用する。

- 鍵は環境変数 `FIELD_ENCRYPTION_KEYS` で管理。バージョン管理付きの Map(`v1` / `v2` ...)
- 現在使用するキーは `FIELD_ENCRYPTION_CURRENT_VERSION`
- 保存形式:`v{n}:base64url(iv ‖ ciphertext+authTag)`
- IV はレコード単位でランダム生成(同じ平文を再保存しても暗号文が変わる)
- Web Crypto API のみを使用(Edge / Node / Browser で同一コード)
- node:crypto は使わない(Edge ランタイム非対応)

実装は [lib/crypto/field-encryption.ts](../../lib/crypto/field-encryption.ts)。

## 結果

得たもの:

- DB ダンプ流出時の被害を最小化(復号には鍵が必要)
- 鍵ローテーション可能(`v2` を追加して current を切り替える)
- 統一インターフェース(`encryptField` / `decryptField`)

諦めたもの / 課題:

- 全文検索が暗号化フィールドに対してできない(構造化フィールドは平文保存)
- 鍵漏洩時は全データ復号可能(クライアントサイド暗号化に移行する余地あり)
- 鍵管理の運用コスト

## 代替案

1. **平文保存**:却下。情報漏洩リスクが高すぎる。
2. **pgcrypto + 共有鍵**:却下。Edge / Browser から扱いにくい、鍵ローテーション運用が難。
3. **クライアントサイド AES**:検討中。パスワード忘れ時のリカバリ機構と組み合わせが必須で、初期実装は重い。サーバーサイドから段階移行する前提。
