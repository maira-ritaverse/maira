# 0006. サーバーサイド暗号化への方針確定(クライアントサイド暗号化を断念)

- ステータス:採用
- 決定日:2026-06-15

## 文脈

[0001 暗号化戦略](./0001-encryption-strategy.md) 採用時、将来的に「クライアントサイド暗号化(ユーザのパスワード由来鍵で暗号化、運営者も復号できない)」へ移行する方向を将来課題として残していた。CLAUDE.md にも差別化要素 #3 として「クライアントサイド暗号化」を掲げていた。

その後、プロダクトの主要機能として以下が計画 / 実装された:

- 履歴書 AI 添削(サーバ側で復号 → Anthropic API へ送信が必須)
- 面接シミュレーター(履歴を暗号化保存、サーバ側で AI と対話)
- **会議録音(Zoom / Google Meet)→ Whisper 文字起こし → Claude による履歴書自動生成**(Phase 1 として着手予定)

これらは「運営者にも復号できない」モデルとは原理的に両立しない:

- ブラウザ内で長時間の音声を Whisper 処理するのは現実的でない
- AI 添削の品質を保つには、サーバ側でクリーン平文を扱う必要がある
- 端末を跨いだセッション継続が困難になる(パスワード変更時に全データロック等)

## 決定

クライアントサイド暗号化への移行を **断念** し、サーバーサイド暗号化(現行)で運用を続ける。

代わりに、ユーザの権利と信頼は以下で担保する:

1. **プライバシーポリシーに保管 / AI 処理範囲を明記**、登録時に同意必須化
2. **運営者によるアクセスを「保管」「本人同意済の AI 処理」「法令対応」の範囲に限定** する旨をポリシーで宣言
3. **暗号化フィールドは引き続き AES-256-GCM で保護**(鍵漏洩リスクを残しつつも、DB ダンプ流出時の被害は最小化)
4. **会議録音のオプトイン同意 UI** を別途用意(機能利用時に追加同意を取る)

## 結果

得たもの:

- 会議録音 → AI 履歴書自動生成 のような高付加価値機能を実装可能に
- AI 添削 / 要約 / 検索 の品質を最大化
- 端末跨ぎ / パスワード復旧の UX を簡素化

諦めたもの:

- 「運営者すらユーザデータを復号できない」という強い保証(マーケティング訴求としても撤回)
- マスターキー復旧コード機構([recovery-code 関連実装]、本決定により削除)
- 一部のプライバシー重視ユーザの取り込み(代替訴求として「プロアクティブ伴走 + 会議録音から履歴書自動生成」で差別化)

## 代替案(再検討するなら)

将来「ハイブリッド」を採る余地はある:

- **二層暗号化**:AI 処理対象は server-side、AI 処理しないフィールドだけ client-side
- 実装は複雑だが、プライバシー訴求と機能性を両立できる
- 必要性が出てきたら 0007 として別 ADR を起こす
