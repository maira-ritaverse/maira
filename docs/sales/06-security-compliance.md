# セキュリティ / コンプライアンス

エージェント業界は個人情報(求職者の経歴・希望条件・身分証等)を大量に扱うため、データ保護は経営の最重要事項です。
Myaira はサーバーサイドの強固な暗号化と RLS による組織完全分離で、業界水準を超える保護を実装しています。

---

## 暗号化方針

### サーバーサイド AES-256-GCM 暗号化

Myaira では機密性の高いフィールドを、サーバーサイドで AES-256-GCM 暗号化してから DB に保存します。
クライアントサイド暗号化(運営者も復号できない方式)は、会議録音・AI 処理機能と両立できないため採用していません。
運営者によるアクセス範囲はプライバシーポリシーで明示し、登録時に同意を取得します。

### 暗号化対象フィールド

| テーブル        | 列                                     | 内容               |
| --------------- | -------------------------------------- | ------------------ |
| messages        | encrypted_content                      | 会話履歴           |
| career_profiles | encrypted_data                         | キャリア棚卸し結果 |
| applications    | encrypted_details                      | 応募情報           |
| agency_tasks    | encrypted_title, encrypted_description | タスク内容         |
| notifications   | encrypted_payload                      | 通知ペイロード     |
| conversations   | encrypted_title                        | 会話タイトル       |
| line_messages   | encrypted_content                      | LINE メッセージ    |
| line_broadcasts | encrypted_body 系                      | LINE 配信本文      |

### 鍵管理

- 環境変数 `FIELD_ENCRYPTION_KEYS`(バージョンマップ)で管理
- `FIELD_ENCRYPTION_CURRENT_VERSION` で現在の鍵バージョンを指定
- 鍵ローテーションをサポート(旧バージョンで暗号化されたデータも復号可能)
- IV はレコード単位でランダム生成
- 保存形式: `v{n}:base64url(iv ‖ ct+authTag)`

### 鍵の保管場所

- サーバーサイド環境変数のみ(Vercel)
- クライアント(ブラウザ)には一切送信されない
- localStorage / sessionStorage / コード への保管禁止

### AI 推論時の取り扱い

- サーバー側で復号 → Anthropic / OpenAI API へ送信
- ブラウザ側に平文を渡さない
- AI 推論結果を再度暗号化して DB に保存
- ストリーミング応答は受信完了時に暗号化保存(中間ストリームは平文 OK)

## RLS(行レベルセキュリティ)

### 全テーブル RLS 有効

全テーブルで Supabase の Row-Level Security を有効化しています。

### 組織分離ポリシー

組織テーブル系には共通して以下のポリシーを適用:

```sql
organization_id = public.current_user_organization_id()
```

= 自分が所属する組織のレコードのみ SELECT / UPDATE / DELETE 可能。
他組織のデータには物理的にアクセス不可です。

### ユーザー個人テーブル

求職者の個人データは `auth.uid() = user_id` で本人のみアクセス可能。

### service_role の例外

Webhook / 内部 cron 等の限定された処理のみ service_role でアクセス。
service_role は環境変数で厳格に管理し、フロントエンドには露出しない設計。

## Webhook 署名検証

外部サービスからの Webhook はすべて HMAC-SHA256 で署名検証:

| サービス           | 検証                    |
| ------------------ | ----------------------- |
| LINE Messaging API | X-Line-Signature ヘッダ |
| Stripe(課金)       | Stripe-Signature ヘッダ |
| Zoom Webhook       | x-zm-signature ヘッダ   |

署名検証に失敗した Webhook は 401 で即拒否。

---

## OAuth / 連携先のスコープ最小化

連携先 API に対しては必要最小限のスコープのみ要求:

| 連携先          | スコープ                                               |
| --------------- | ------------------------------------------------------ |
| Google Calendar | `https://www.googleapis.com/auth/calendar.events` のみ |
| Google ID       | `openid`, `email`                                      |
| Zoom            | `meeting:write` 系のみ                                 |
| LINE            | Messaging API(個別チャネル単位)                        |

不要な広い権限(例: Google Drive へのフルアクセス)は一切要求しません。

## プライバシーポリシー

### 運営者の閲覧範囲

プライバシーポリシーで以下に明示限定:

1. **保管** — DB に暗号化して保存する範囲
2. **本人同意済みの AI 処理** — Anthropic / OpenAI への送信は同意を取得済みの範囲のみ
3. **法令対応** — 警察・裁判所からの正当な令状があった場合のみ

### 登録時の同意取得

- サインアップ時にプライバシーポリシーへの同意を必須化
- 重要改訂時はプライバシーポリシーモーダルで再同意を取得

### データ取り扱いの透明性

- AI 処理を行う機能には、UI 上で「外部 AI サービスに送信されます」の明示
- 求職者のデータが何の AI で何の処理に使われるかを文書化(プライバシーポリシー第5条)

## 監査ログ

組織内の重要な操作は監査ログに自動記録:

| 操作                      | 記録内容                             |
| ------------------------- | ------------------------------------ |
| メンバー招待              | 招待者 / 招待対象 / 役割 / 日時      |
| 役割変更(admin ↔ advisor) | 変更者 / 対象 / 変更前後 / 日時      |
| メンバー削除              | 操作者 / 対象 / 日時                 |
| 権限フラグ変更            | 操作者 / 対象 / 権限キー / 値 / 日時 |
| クライアント編集          | 編集者 / 対象 / 変更内容 / 日時      |

### 監査ログの保管

- DB 内の `member_audit_log` 系テーブルで永続保存
- admin は `/agency/members` 内で閲覧可能
- 運営側も `/admin/audit-logs` で全組織横断閲覧可能

## バックアップ

### Supabase 標準バックアップ

- 日次の自動バックアップ
- Point-in-Time Recovery(直近の任意時点に復旧可能)

### データエクスポート

- 解約時に CSV / JSON で全データをエクスポート(運営に依頼)
- 進行中の組織は admin 画面から個別テーブルをエクスポート可能(求職者 / 求人 / 応募)

---

## 解約・データ削除

### 解約時のデータ取り扱い

1. 解約予約 → 期末まで利用可能
2. 期末以降は読み取り専用
3. 30日間データ保持(再契約時に復元可能)
4. 30日経過後に完全削除

### 求職者個人からの削除依頼

- 求職者本人がアプリ内から「アカウント削除」を実行可能
- 削除後は復元不可
- エージェントとの紐付け(連携)は解除され、エージェント側からも閲覧不可になる

## コンプライアンス対応

### 個人情報保護法

- 取得・利用・第三者提供のすべてで適法な根拠を確保
- 利用目的の明示(プライバシーポリシー)
- 開示・訂正・削除請求への対応窓口を設置(support@maira.pro)

### 職業安定法 / 労働者派遣法

- 求人情報の真実性に関するエージェント側の責任を明示
- 求職者の同意なき情報提供を禁止

### GDPR(EU 圏ユーザー対応)

- 現在は国内事業のみだが、将来の海外展開を想定して設計
- データ削除権 / 移動権を機能として実装済(エクスポート / 完全削除)

### SOC 2 / ISO 27001 等

- 現状未取得だが、将来取得を視野に設計
- 監査ログ / バックアップ / RBAC / 暗号化など基礎要件は実装済

## インシデント対応

### 通常時

- セキュリティパッチを継続的に適用(Next.js / Supabase / 各 SDK)
- 依存ライブラリの脆弱性スキャン(GitHub Dependabot)
- 内部ペネトレーションテスト(年次)

### 万一インシデントが発生した場合

1. 影響範囲の特定(対象テーブル / 期間 / 影響ユーザー数)
2. 当該データを利用停止
3. 個別通知(影響を受けた組織 / ユーザー宛にメール)
4. 個人情報保護委員会への報告(法令の閾値を超える場合)
5. 原因分析 + 再発防止策の実装

## よくあるセキュリティ質問への回答

### Q. 運営者は私たちのクライアント情報を見られますか?

A. 暗号化された状態で DB に保存されているため、運営者であってもアプリ機能を介さない限り平文は見られません。
ただし「保管・AI 処理・法令対応」の3範囲に限定して、暗号化キーへのアクセスは持っています。
これはプライバシーポリシーで明示しています。

### Q. AI に送ったデータは AI 提供企業に保存されますか?

A. Anthropic / OpenAI ともに、API 経由のデータを学習に使用しない設定で利用しています(Enterprise / Business 規約)。
データの一時保存は 30日以内で削除されます(各社の標準ポリシー)。

### Q. 他社の求職者データが見える可能性はありますか?

A. RLS により物理的に不可能です。仮にアプリにバグがあっても、DB レベルで他組織のデータは取得できません。

### Q. 退会後のデータはどうなりますか?

A. 30日間の保持期間後、完全削除されます。CSV / JSON でのエクスポートも事前に可能です。

### Q. パスワード漏洩時の影響は?

A. パスワードは bcrypt(salt付き)でハッシュ化保存、復号不可能。万一漏洩しても元のパスワードに戻せません。
ログイン試行レート制限 + Supabase Auth の標準セキュリティ機能を併用。

### Q. データセンターの場所は?

A. Supabase Northeast Asia (Tokyo) リージョン。国内法の適用範囲内です。
